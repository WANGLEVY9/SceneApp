# 原型文档

## 阶段 0：需求澄清与信息架构
- 目标：覆盖游客导览 + 打卡 + 礼品兑换的闭环，管理员具备内容运营能力。
- 产出：信息架构草图（白板），定义用户端的首页、打卡、礼品、个人中心；后台的登录、仪表盘、景点/商家/打卡点、礼品、兑换记录。

## 阶段 1：低保真（Lo-Fi）线框
- 工具：手绘。
- 重点决策：
  - 用户端底部导航（首页/打卡/礼品/我的）。
  - 打卡流程分步（选择点位 → 触发 → 成功反馈）。
  - 后台导航以左侧菜单 + 顶部面包屑，优先 CRUD 功能。
- 反馈：确认页面数量、导航层级、主要字段（景点名称、坐标、描述、积分等）。

## 阶段 2：中保真（Mid-Fi）交互稿
- 工具：Figma。
- 用户端：
  - 首页卡片化展示景点与商家，增设“推荐”分区，预留动态流入口。
  - 打卡页面增加地理信息提示、成功动画占位。
  - 礼品列表以卡片 + 兑换按钮形式，兑换记录为列表。
- 管理端：
  - 登录页简洁表单；
  - 仪表盘提供数据概览占位（后续可接入看板）；
  - 列表支持搜索/分页，表单分组展示核心字段。
- 反馈：交互流程通过走查，确定 API 字段对齐。

## 阶段 3：高保真（Hi-Fi）视觉稿
- 用户端：
  - 应用 HarmonyOS 设计规范，统一主色调与卡片阴影；
  - 礼品页加入积分徽标；
  - 动态流卡片预留图片/文本区，点赞/评论操作位。
- 管理端：
  - 采用轻量配色，突出表格与按钮；
  - 登录后默认跳转仪表盘；
  - 列表页支持“新增/编辑/删除”按钮及状态标签。
- 反馈：视觉确认后进入前端开发；部分视觉细节在实现中按组件库能力微调。

## 阶段 4：实现对齐与走查
- 用户端：ArkTS 页面与 API 对接，Http 工具支持 token、错误提示；实机/模拟器走查导航、打卡、礼品兑换流。
- 管理端：Vue 路由与 Axios 拦截器联调，确保表格 CRUD、状态更新与后端一致。
- 迭代记录：
  - 打卡成功反馈由模态改为顶部提示，减少打断。
  - 管理端新增兑换记录状态更新入口，以满足运营闭环。

## 阶段 5：预留升级与扩展
- 推荐/个性化：预留排序/地理参数，未来可接入 LLM/大数据推荐。
- 数据看板：仪表盘支持扩展折线/柱状组件，展示打卡量、兑换量、活跃度。
- 多端适配：管理端支持移动端响应式布局的占位样式；用户端预留国际化与夜间模式样式变量。
